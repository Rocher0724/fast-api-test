# 1장. 에러는 어디에서 나는가? (10분)

## 학습 목표

에러가 발생할 수 있는 전체 그림을 이해하고, 에러를 만났을 때 "어디를 봐야 하는지" 감을 잡는다.

---

## 1.1 웹 서비스의 전체 구조 이해

웹 서비스는 여러 구간이 연결되어 동작한다. 에러를 잡으려면 먼저 이 전체 흐름을 이해해야 한다.

```
[사용자의 브라우저]  ←→  [인터넷]  ←→  [서버 (FastAPI)]  ←→  [데이터베이스]
   (프론트엔드)                          (백엔드)              (DB)
   HTML/CSS/JS                         Python 코드           데이터 저장소
```

### 각 구간이 하는 일

| 구간 | 역할 | 비유 |
|------|------|------|
| 브라우저 (프론트엔드) | 사용자가 보는 화면, 버튼 클릭 처리 | 식당의 메뉴판과 주문 벨 |
| 인터넷 (네트워크) | 브라우저와 서버 사이의 통신 | 주문을 전달하는 웨이터 |
| 서버 (백엔드) | 요청을 받아서 처리하는 로직 | 주방에서 요리하는 셰프 |
| 데이터베이스 | 데이터를 저장하고 꺼내주는 곳 | 식재료 창고 |

### 왜 이걸 알아야 하는가?

에러가 났을 때 **"어디서 난 에러인지"**를 모르면 엉뚱한 곳을 고치게 된다.

예를 들어:
- 화면에 데이터가 안 나온다 → 프론트 문제? 서버 문제? DB 문제?
- "서버가 응답하지 않습니다" → 서버가 죽은 건지, 네트워크 문제인지, 배포 설정 문제인지?

**에러를 고치는 첫 번째 단계는 항상 "어디서 난 에러인가?"를 파악하는 것이다.**

## 1.2 에러의 5가지 발생 지점

```
① 프론트엔드 에러        → 브라우저 개발자 도구(DevTools)에서 확인
② 백엔드 에러            → 서버 터미널 로그에서 확인
③ 서버↔DB 에러          → 서버 로그 + DB 연결 상태에서 확인
④ 프론트↔백 통신 에러    → 브라우저 Network 탭 + 서버 로그 양쪽에서 확인
⑤ 배포 환경 에러         → Render.com 대시보드 로그에서 확인
```

### ① 프론트엔드 에러

**어디서 확인?** 브라우저 개발자 도구 (F12) → Console 탭

발생 예시:
- 버튼을 눌렀는데 아무 반응이 없다
- 화면 레이아웃이 깨진다
- JavaScript 코드에 오타가 있다

### ② 백엔드 에러

**어디서 확인?** 서버를 실행한 터미널 창

발생 예시:
- API를 호출했는데 500 에러가 돌아온다
- 서버가 시작되자마자 죽는다
- 특정 기능만 에러가 난다

### ③ 서버↔DB 에러

**어디서 확인?** 서버 터미널 로그 (DB 관련 에러 메시지)

발생 예시:
- 데이터를 저장했는데 안 들어간다
- "connection refused" 에러가 난다
- 서버가 갑자기 느려진다 (DB 연결 문제)

### ④ 프론트↔백 통신 에러

**어디서 확인?** 브라우저 Network 탭 + Console 탭

발생 예시:
- CORS 에러 (빨간 글씨로 "blocked by CORS policy")
- API 주소를 잘못 적었다
- 서버가 기대하는 데이터 형식과 다르게 보냈다

### ⑤ 배포 환경 에러

**어디서 확인?** Render.com 대시보드 → Logs 탭

발생 예시:
- 로컬에서는 잘 되는데 배포하면 안 된다
- 배포할 때 빌드 에러가 난다
- 환경변수를 설정하지 않았다

## 1.3 HTTP 상태 코드 빠르게 읽기

브라우저와 서버는 **HTTP 상태 코드**라는 숫자로 대화한다. 이 숫자만 읽을 줄 알아도 에러의 원인을 절반은 파악할 수 있다.

### 외워야 할 핵심 코드

| 코드 | 의미 | 누구 잘못? | 자주 만나는 상황 |
|------|------|-----------|----------------|
| **200** | 성공 | - | 정상 동작 |
| **201** | 생성 성공 | - | 데이터 저장 성공 |
| **400** | 잘못된 요청 | 프론트 | 보낸 데이터 형식이 이상함 |
| **401** | 인증 실패 | 프론트 | 로그인 안 했거나 토큰 만료 |
| **403** | 권한 없음 | 프론트 | 로그인은 했는데 권한이 없음 |
| **404** | 못 찾음 | 프론트 or 백 | URL 경로가 틀림 |
| **405** | 메서드 불허 | 프론트 | GET으로 보냈는데 POST만 받음 |
| **422** | 처리 불가 | 프론트 | FastAPI가 데이터 검증 실패 |
| **500** | 서버 내부 에러 | 백엔드 | 서버 코드에 버그가 있음 |
| **502/503** | 서버 다운 | 배포/인프라 | 서버가 죽었거나 과부하 |

### 쉽게 외우는 법

```
2xx = 성공 👍 "다 잘 됐어"
4xx = 클라이언트(프론트) 잘못 👈 "네가 잘못 보냈어"
5xx = 서버(백엔드) 잘못 👉 "내(서버)가 잘못했어"
```

### 실전 팁

> 에러가 났을 때 가장 먼저 볼 것: **상태 코드가 4xx인지 5xx인지**
> - 4xx → 프론트엔드 코드를 확인
> - 5xx → 백엔드 코드를 확인
> - 이것만으로도 디버깅 시간을 절반으로 줄일 수 있다

---

## 핵심 정리

1. 에러가 나면 **"어디서 난 에러인가?"**를 가장 먼저 파악한다
2. 확인할 곳은 크게 5군데: 브라우저 Console, 브라우저 Network, 서버 터미널, DB 로그, 배포 로그
3. HTTP 상태 코드의 첫 자리 숫자(2/4/5)만 봐도 원인의 방향을 알 수 있다
