# 2장. 프론트엔드 에러 확인법: 브라우저 개발자 도구

## 학습 목표

브라우저에서 에러가 났을 때 **개발자 도구(DevTools)**를 열어 문제를 찾는 방법을 익힌다.

---

## 2.1 개발자 도구(DevTools) 열기

### 여는 방법

| 브라우저 | 단축키 (Mac) | 단축키 (Windows) |
|---------|-------------|-----------------|
| Chrome  | `Cmd + Option + I` | `F12` 또는 `Ctrl + Shift + I` |
| Safari  | `Cmd + Option + I` | - |
| Firefox | `Cmd + Option + I` | `F12` |

또는 화면에서 **마우스 우클릭 → "검사(Inspect)"** 를 누르면 된다.

### 꼭 알아야 할 탭 3개

DevTools에는 탭이 많지만, 디버깅에 필수인 건 3개뿐이다.

```
┌──────────┬──────────┬──────────┬─────────┐
│ Elements │ Console  │ Network  │ ...     │
└──────────┴──────────┴──────────┴─────────┘
     ①          ②          ③
```

| 탭 | 용도 | 언제 보는가 |
|----|------|-----------|
| **Elements** | HTML 구조와 CSS 스타일 확인 | 화면이 깨지거나 레이아웃이 이상할 때 |
| **Console** | JavaScript 에러 메시지 확인 | 버튼이 안 먹거나 기능이 안 될 때 |
| **Network** | API 요청/응답 확인 | 서버와의 통신이 실패할 때 |

---

## 2.2 Console 탭 - JS 에러 잡기

### 에러 메시지 읽는 법

Console 탭을 열면 **빨간색 글씨**로 에러가 표시된다. 에러 메시지는 보통 이런 구조다:

```
❌ Uncaught TypeError: Cannot read properties of undefined (reading 'name')
       ───────────          ─────────────────────────────────────────────
       에러 종류                        에러 설명
   at script.js:15:3
   ─────────────────
   파일명:줄번호:열번호
```

**읽는 순서:**
1. **에러 종류** - 어떤 종류의 에러인지 (TypeError, ReferenceError 등)
2. **에러 설명** - 구체적으로 뭐가 잘못됐는지
3. **파일명:줄번호** - 에러가 발생한 위치 → **이걸 클릭하면 해당 코드로 이동한다**

### 자주 만나는 에러 유형

#### `Uncaught ReferenceError: xxx is not defined`

```javascript
// ❌ 에러: userName이 정의되지 않았음
console.log(userName);

// ✅ 해결: 변수를 먼저 선언
const userName = "홍길동";
console.log(userName);
```

**원인:** 변수명 오타이거나, 변수를 선언하기 전에 사용했다.

#### `Uncaught TypeError: Cannot read properties of undefined`

```javascript
// ❌ 에러: data가 undefined인데 .name을 읽으려 함
const data = undefined;
console.log(data.name);

// ✅ 해결: data가 있는지 먼저 확인
if (data) {
    console.log(data.name);
}
```

**원인:** API 응답이 아직 안 왔는데 데이터를 사용하려 했거나, 데이터 구조를 잘못 알고 있다.

#### `Uncaught SyntaxError: Unexpected token`

```javascript
// ❌ 에러: JSON 형식이 잘못됨
const data = JSON.parse("{ name: '홍길동' }");

// ✅ 해결: JSON은 반드시 큰따옴표 사용
const data = JSON.parse('{ "name": "홍길동" }');
```

**원인:** JavaScript 문법 오류. 괄호 안 닫았거나, 쉼표 빠뜨렸거나, JSON 형식이 틀렸다.

### console.log() 디버깅

에러 메시지만으로 원인을 모를 때, 의심되는 곳에 `console.log()`를 넣어서 값을 확인한다.

```javascript
async function loadUsers() {
    console.log("1. 함수 시작");            // 함수가 실행되는지 확인

    const response = await fetch("/api/users");
    console.log("2. 응답 상태:", response.status);  // 응답 코드 확인

    const data = await response.json();
    console.log("3. 받은 데이터:", data);    // 실제 데이터 확인

    // 여기서 에러가 나면 data의 구조가 예상과 다른 것
    document.getElementById("user-name").textContent = data.name;
}
```

> **팁:** console.log에 **번호를 매기면** 어디까지 실행되고 어디서 멈추는지 바로 알 수 있다.

## 2.3 Network 탭 - API 통신 에러 잡기

### Network 탭이 중요한 이유

프론트엔드가 서버에 데이터를 요청할 때, 그 요청과 응답의 **모든 내용**이 Network 탭에 기록된다. "서버에서 에러가 난다"고 생각했는데, 실은 **요청 자체를 잘못 보낸 것**인 경우가 매우 많다.

### 확인하는 방법

1. DevTools → **Network 탭** 클릭
2. 화면에서 에러가 나는 동작을 수행 (버튼 클릭 등)
3. 목록에 새로 뜨는 요청을 클릭

### 요청 하나를 클릭하면 보이는 정보

```
┌─ General ─────────────────────────────────────┐
│ Request URL:    https://myapp.onrender.com/api/users
│ Request Method: POST                          │
│ Status Code:    422 Unprocessable Entity       │ ← 여기!
└───────────────────────────────────────────────┘

┌─ Request Headers ─────────────────────────────┐
│ Content-Type: application/json                │
└───────────────────────────────────────────────┘

┌─ Request Payload (내가 보낸 데이터) ───────────┐
│ { "user_name": "홍길동" }                      │ ← 내가 보낸 것
└───────────────────────────────────────────────┘

┌─ Response (서버가 보낸 응답) ──────────────────┐
│ { "detail": [{ "msg": "field required",       │ ← 서버가 뭐가 잘못됐는지 알려줌
│    "type": "value_error.missing",             │
│    "loc": ["body", "username"] }] }           │ ← "username" 필드가 필요하다!
└───────────────────────────────────────────────┘
```

### 체크리스트: Network 탭에서 확인할 것

| 확인 항목 | 어디서 보는가 | 무엇을 확인하는가 |
|----------|-------------|-----------------|
| **Status Code** | 목록의 Status 열 | 200이 아니면 문제 있음 |
| **Request URL** | 요청 상세 → General | URL 경로가 맞는지, 오타 없는지 |
| **Request Method** | 요청 상세 → General | GET/POST/PUT/DELETE가 맞는지 |
| **Request Payload** | 요청 상세 → Payload 탭 | 보내는 데이터 형식이 맞는지 |
| **Response** | 요청 상세 → Response 탭 | 서버가 뭐라고 했는지 에러 메시지 확인 |

### 빨간색 요청 = 실패한 요청

Network 탭에서 **빨간색**으로 표시되는 요청은 실패한 것이다.

- **빨간색이 아예 없다** → 요청 자체가 안 나간 것 (JS 코드 문제)
- **빨간색 + CORS error** → 4장에서 다룸
- **빨간색 + 4xx/5xx** → 요청은 나갔지만 서버가 에러를 응답

### 알아두면 좋은 지식: 요청이 기록 안 될 때

Network 탭을 열기 **전에** 발생한 요청은 기록되지 않는다. 에러를 재현할 때는:

1. 먼저 Network 탭을 연다
2. 왼쪽 상단의 빨간 동그라미(●)가 활성화되어 있는지 확인
3. 그 다음에 에러가 나는 동작을 수행

> **팁:** "Preserve log" 체크박스를 켜면 페이지가 새로고침되어도 기록이 유지된다.

---

## 핵심 정리

1. **화면이 이상하면** → Elements 탭에서 HTML/CSS 확인
2. **기능이 안 되면** → Console 탭에서 빨간 에러 메시지 확인
3. **서버 통신이 안 되면** → Network 탭에서 요청/응답 확인
4. 에러 메시지의 **파일명:줄번호**를 클릭하면 문제 코드로 바로 갈 수 있다
5. `console.log()`에 번호를 매겨서 "어디까지 실행되는지" 확인하는 것이 가장 기본적인 디버깅
