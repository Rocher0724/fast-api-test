FastAPI 서버에서 에러가 발생했을 때 **어디서 확인하고**, **어떻게 읽고**, **어떤 도구로 테스트하는지** 배운다.

## 3.1 FastAPI 서버 로그 읽기

### 로그란?

서버가 실행되면서 일어나는 일을 텍스트로 기록한 것이다. 식당으로 비유하면 주방 CCTV 같은 것이다. 에러가 나면 **로그를 보고** 무슨 일이 있었는지 파악한다.

### 어디서 보는가?

**로컬 개발 시:** 서버를 실행한 터미널 창

```bash
# 서버를 실행하면 이런 로그가 나온다
$ uvicorn app.main:app --reload
INFO:     Uvicorn running on http://127.0.0.1:8000
INFO:     Started reloader process [12345]
INFO:     Started server process [12346]
INFO:     Waiting for application startup.
INFO:     Application startup complete.

# 요청이 들어올 때마다 로그가 찍힌다
INFO:     127.0.0.1:52341 - "GET /health HTTP/1.1" 200 OK
INFO:     127.0.0.1:52342 - "GET /api/users HTTP/1.1" 500 Internal Server Error
#                                                       ─── 이 숫자를 확인!
```

**배포 환경 (Render.com):** 대시보드 → Logs 탭 (6장에서 자세히 다룸)

**실제 클라우드 서버 (AWS, GCP 등):** SSH로 서버에 접속해서 로그 확인

```bash
# 실무에서는 이런 식으로 서버에 직접 접속해서 로그를 본다
$ ssh user@my-server-ip
$ tail -f /var/log/app/server.log    # 실시간 로그 보기
$ cat /var/log/app/server.log | grep ERROR   # 에러만 필터링
```

> **알아두면 좋은 것:** Render.com은 이걸 웹 대시보드에서 편하게 볼 수 있게 해주는 것이다. 하지만 나중에 AWS나 직접 서버를 운영하게 되면 SSH로 접속해서 로그 파일을 직접 읽어야 한다. 원리는 같다 — **"서버가 남긴 기록을 읽는다."**

### Traceback(역추적) 읽는 법

서버에서 에러가 나면 이런 긴 메시지가 나온다. 이걸 **Traceback**이라고 한다.

```
ERROR:    Exception in ASGI application
Traceback (most recent call last):
  File "/app/venv/lib/python3.11/site-packages/uvicorn/protocols/http/h11_impl.py", line 373
    ...
  File "/app/main.py", line 25, in get_users        ← ② 에러가 발생한 내 코드
    result = db.execute(query)
  File "/app/database.py", line 12, in execute       ← ① 실제 에러가 터진 곳
    raise ConnectionError("Database connection failed")
ConnectionError: Database connection failed           ← ③ 에러 메시지 (가장 중요!)
```

**읽는 순서: 아래에서 위로!**

```
③ 맨 아래: 에러 메시지 → "무슨 에러인지" (가장 먼저 읽는다)
② 중간: 내 코드 파일 → "내 코드 어디에서 에러가 났는지" (파일명:줄번호)
① 위쪽: 프레임워크 내부 코드 → (보통 무시해도 됨)
```

> **핵심:** Traceback에서 **맨 아래 줄**과 **내가 작성한 파일이 나오는 줄**만 보면 된다. 나머지는 FastAPI 내부 코드이므로 처음엔 무시해도 괜찮다.

## 3.2 FastAPI가 자주 뱉는 에러들

### 422 Unprocessable Entity

**가장 자주 만나는 에러.** FastAPI는 데이터 검증을 자동으로 해주는데, 요청 데이터가 기대와 다르면 422를 뱉는다.

```python
# 서버 코드: username(문자열)과 age(숫자)를 기대
from pydantic import BaseModel

class UserCreate(BaseModel):
    username: str
    age: int

@app.post("/api/users")
def create_user(user: UserCreate):
    return {"message": f"{user.username} 생성 완료"}
```

```javascript
// ❌ 프론트에서 이렇게 보내면 422 에러
fetch("/api/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
        user_name: "홍길동",  // ← 'username'이어야 하는데 'user_name'으로 보냄
        age: "스물다섯"       // ← 숫자여야 하는데 문자열로 보냄
    })
});
```

**에러 응답:**
```json
{
    "detail": [
        {
            "loc": ["body", "username"],
            "msg": "field required",
            "type": "value_error.missing"
        },
        {
            "loc": ["body", "age"],
            "msg": "value is not a valid integer",
            "type": "type_error.integer"
        }
    ]
}
```

**확인 방법:** 브라우저 Network 탭 → Response에서 `detail`을 읽으면 **어떤 필드가 잘못됐는지** 정확히 알려준다.

### 500 Internal Server Error

서버 코드 내부에서 예외(Exception)가 발생한 것이다. 코드에 버그가 있다는 뜻.

```python
@app.get("/api/users/{user_id}")
def get_user(user_id: int):
    users = [{"id": 1, "name": "홍길동"}]
    # ❌ user_id가 2이면 IndexError 발생 → 500 에러
    return users[user_id]
```

**확인 방법:** 서버 터미널에 Traceback이 출력된다. **맨 아래 줄을 읽자.**

### 404 Not Found

요청한 URL 경로에 해당하는 API가 없다.

```python
# 서버에 등록된 경로
@app.get("/api/users")     # ← 이 경로만 존재

# 프론트에서 요청한 경로
fetch("/api/user")          # ← 's'가 빠짐 → 404
fetch("/api/Users")         # ← 대문자 → 404 (URL은 대소문자 구분)
```

**확인 방법:** 브라우저 Network 탭에서 요청 URL을 확인하고, 서버 코드의 라우트 경로와 비교한다.

### ImportError / ModuleNotFoundError

패키지를 설치하지 않았거나 import 경로가 틀렸을 때.

```
ModuleNotFoundError: No module named 'sqlalchemy'
```

**확인 방법:** 서버 시작 시 터미널에서 바로 에러가 나온다. `pip install 패키지명`으로 해결.

## 3.3 /docs (Swagger UI)로 API 테스트하기

### Swagger UI란?

FastAPI는 **자동으로 API 테스트 페이지**를 만들어준다. 서버를 실행하고 브라우저에서 접속:

```
http://localhost:8000/docs
```

### 왜 중요한가?

**프론트엔드 없이 API만 따로 테스트**할 수 있다. 이것으로 에러의 원인을 분리한다:

```
문제: 화면에서 데이터가 안 나온다

→ /docs에서 직접 API 호출해본다
  → /docs에서도 에러 → 백엔드 문제 (서버 코드를 고쳐야 함)
  → /docs에서는 정상 → 프론트엔드 문제 (JS 코드를 고쳐야 함)
```

### 사용 방법

1. `/docs` 접속
2. 테스트할 API 클릭 → **"Try it out"** 버튼 클릭
3. 필요한 파라미터 입력
4. **"Execute"** 클릭
5. 아래에 나오는 **Response body**와 **Status code** 확인

> **실전 팁:** 에러가 났을 때 "/docs에서 테스트해봤어요" 라고 말할 수 있으면, 프론트/백 어디가 문제인지를 이미 분리한 것이다. 이것만으로도 디버깅 능력이 크게 올라간다.

## 3.4 print() 디버깅과 에러 처리

### print()로 값 추적하기

서버 코드에서도 `print()`는 강력한 디버깅 도구다.

```python
@app.post("/api/users")
def create_user(user: UserCreate):
    print("=== 요청 받음 ===")
    print(f"받은 데이터: {user}")
    print(f"username: {user.username}")
    print(f"age: {user.age}")

    # DB 저장 로직
    result = save_to_db(user)
    print(f"DB 저장 결과: {result}")

    return {"message": "성공"}
```

`print()`의 출력은 **서버를 실행한 터미널**에 나온다. 브라우저에는 안 보인다.

### FastAPI에서 에러를 발생시키는 방법

FastAPI에서는 `HTTPException`을 사용해서 **의도적으로** 에러를 만들 수 있다. 이건 버그가 아니라, "이 상황에서는 에러를 응답해야 한다"는 의미다.

```python
from fastapi import FastAPI, HTTPException

@app.get("/api/users/{user_id}")
def get_user(user_id: int):
    user = find_user_by_id(user_id)  # DB에서 사용자 검색

    if user is None:
        # 사용자가 없으면 404 에러를 의도적으로 발생시킨다
        raise HTTPException(
            status_code=404,
            detail=f"ID {user_id}인 사용자를 찾을 수 없습니다"
        )

    return user
```

**왜 에러를 "발생시키는" 건가?**

에러 처리를 안 하면 서버가 어떤 에러든 `500 Internal Server Error`로 응답한다. 하지만 `HTTPException`으로 **구체적인 에러 코드와 메시지**를 보내면:
- 프론트엔드가 에러 종류를 구분해서 적절한 메시지를 보여줄 수 있다
- 디버깅할 때 뭐가 잘못됐는지 바로 알 수 있다

```python
# ❌ 에러 처리 안 하면: 500 + 알 수 없는 에러
# ✅ 에러 처리 하면: 404 + "ID 5인 사용자를 찾을 수 없습니다"
```

### try-except로 예상치 못한 에러 잡기

```python
@app.post("/api/users")
def create_user(user: UserCreate):
    try:
        save_to_db(user)
        return {"message": "성공"}
    except Exception as e:
        # 에러가 나도 서버가 죽지 않고, 에러 내용을 로그로 남긴다
        print(f"에러 발생: {e}")
        raise HTTPException(status_code=500, detail="사용자 생성 중 오류가 발생했습니다")
```

> **알아두면 좋은 것:** 실무에서는 `print()` 대신 Python의 `logging` 모듈을 사용한다. 로그 레벨(DEBUG, INFO, WARNING, ERROR)을 나눌 수 있고, 파일에 자동 저장할 수 있기 때문이다. 하지만 시작은 `print()`로 충분하다.

## 핵심 정리

1. **에러 확인 순서:** 서버 터미널 → Traceback 맨 아래 줄 → 내 파일:줄번호
2. **422 에러**는 Response의 `detail`을 읽으면 어떤 필드가 잘못됐는지 알 수 있다
3. **500 에러**는 서버 터미널의 Traceback을 읽어야 한다
4. **/docs**에서 API를 테스트하면 프론트 문제인지 백엔드 문제인지 분리할 수 있다
5. `HTTPException`으로 **의도적인 에러 응답**을 만들어야 디버깅이 쉬워진다
6. 실제 운영 서버에서는 SSH 접속이나 로그 서비스를 통해 로그를 확인한다
