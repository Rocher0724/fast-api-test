# 4장. 서버↔DB 에러: 데이터베이스 통신 문제

## 학습 목표

서버가 데이터베이스와 통신할 때 발생하는 에러를 **어디서 확인하고**, 원인을 파악하는 방법을 배운다. 데이터베이스의 기본 동작 원리도 함께 이해한다.

---

## 4.1 DB 연결 에러

### 에러는 어디서 확인하는가?

**서버 터미널 (또는 Render Logs)**에서 확인한다. DB 에러는 프론트엔드에서는 보통 `500 Internal Server Error`로만 보이기 때문에, **반드시 서버 로그를 열어야 진짜 원인을 알 수 있다.**

```
# 브라우저에서 보이는 것 (정보 부족)
500 Internal Server Error

# 서버 터미널에서 보이는 것 (진짜 원인)
sqlalchemy.exc.OperationalError: (psycopg2.OperationalError)
    could not connect to server: Connection refused
    Is the server running on host "localhost" and accepting TCP/IP connections on port 5432?
```

### 자주 만나는 연결 에러들

#### `Connection refused` - DB 서버가 꺼져있거나 주소가 틀림

```
원인:
- DB 서버가 실행되고 있지 않다
- 호스트 주소나 포트 번호가 틀렸다
- 로컬에서는 localhost인데 배포에서는 클라우드 DB 주소를 써야 한다

확인 방법:
1. 서버 터미널에서 에러 메시지 확인
2. DATABASE_URL 환경변수 값 확인
3. DB 서버가 실행 중인지 확인
```

#### 연결 문자열(DATABASE_URL) 문제

DATABASE_URL은 서버가 DB에 접속하기 위한 "주소"다. 이 형식이 자주 틀린다.

```
# DATABASE_URL 형식
postgresql://유저명:비밀번호@호스트:포트/데이터베이스명

# 예시
postgresql://myuser:mypassword@db-server.render.com:5432/mydb

# 자주 하는 실수들
postgresql://myuser:mypassword@localhost:5432/mydb     ← 배포 시 localhost 쓰면 안 됨
postgresql://myuser:mypasswrd@db-server:5432/mydb      ← 비밀번호 오타
postgres://myuser:mypassword@db-server:5432/mydb       ← 'postgresql'이 아닌 'postgres' (라이브러리에 따라 다름)
```

#### 연결 타임아웃

```
sqlalchemy.exc.OperationalError: (psycopg2.OperationalError)
    connection timed out

원인:
- DB 서버가 너무 멀리 있거나 응답이 느리다
- 방화벽/보안그룹이 연결을 막고 있다
- DB 서버에 과부하가 걸렸다
```

## 4.2 SQL / ORM 쿼리 에러

### ORM이란?

Python 코드로 DB를 다루게 해주는 도구다. SQL 쿼리를 직접 안 쓰고도 데이터를 저장/조회할 수 있다. FastAPI에서는 보통 **SQLAlchemy**를 사용한다.

```python
# SQL로 직접 쓰면
# SELECT * FROM users WHERE id = 1

# ORM (SQLAlchemy)으로 쓰면
user = db.query(User).filter(User.id == 1).first()
```

### 자주 만나는 쿼리 에러들

#### 테이블이 존재하지 않음

```
sqlalchemy.exc.ProgrammingError:
    (psycopg2.errors.UndefinedTable) relation "users" does not exist
```

**원인:** 코드에서는 `users` 테이블을 사용하는데, DB에 실제로 그 테이블이 없다.

**해결:**
```bash
# 마이그레이션을 실행해서 테이블을 만든다
# Alembic을 사용하는 경우:
alembic upgrade head

# 또는 직접 테이블을 생성하는 코드를 실행:
python -c "from app.database import Base, engine; Base.metadata.create_all(bind=engine)"
```

#### 컬럼명 불일치

```python
# 코드에서 정의한 모델
class User(Base):
    __tablename__ = "users"
    user_name = Column(String)     # ← 코드에서는 'user_name'

# 하지만 DB에 실제 컬럼이 'username'이면 에러
```

**확인 방법:** DB에 직접 접속해서 실제 테이블 구조를 확인한다.

#### 타입 불일치

```
sqlalchemy.exc.DataError:
    (psycopg2.errors.InvalidTextRepresentation)
    invalid input syntax for type integer: "스물다섯"
```

**원인:** 숫자 컬럼에 문자열을 넣으려 했다.

#### IntegrityError - 제약조건 위반

```
sqlalchemy.exc.IntegrityError:
    (psycopg2.errors.UniqueViolation)
    duplicate key value violates unique constraint "users_email_key"
    DETAIL: Key (email)=(test@example.com) already exists.
```

**원인:** 이미 같은 이메일로 가입한 사용자가 있는데 또 만들려고 했다 (unique 제약조건 위반).

```
sqlalchemy.exc.IntegrityError:
    (psycopg2.errors.NotNullViolation)
    null value in column "username" violates not-null constraint
```

**원인:** 필수 필드(NOT NULL)에 값을 안 넣었다.

## 4.3 커넥션 풀(Connection Pool) 이해하기

### 커넥션 풀이란?

DB에 연결하는 것은 시간이 걸리는 작업이다. 그래서 서버는 **미리 여러 개의 연결을 만들어 놓고 돌려쓴다.** 이걸 **커넥션 풀(Connection Pool)**이라고 한다.

```
                    커넥션 풀 (예: 최대 10개)
                  ┌──────────────────────┐
  요청 A ────→    │  연결1 ──→ DB        │
  요청 B ────→    │  연결2 ──→ DB        │
  요청 C ────→    │  연결3 ──→ DB        │
  요청 D ────→    │  ...                 │
  요청 E ────→    │  연결10 ──→ DB       │
  요청 F ────→    │  (대기...)           │ ← 연결이 다 차면 대기하다가 타임아웃
                  └──────────────────────┘
```

### 커넥션 풀 부족 에러

```
sqlalchemy.exc.TimeoutError:
    QueuePool limit of size 5 overflow 10 reached,
    connection timed out, timeout 30
```

**원인:** 동시에 너무 많은 요청이 들어오거나, 연결을 사용한 뒤 반환하지 않았다.

**왜 이게 중요한가?**

- Render.com 무료 DB는 보통 **연결 수 제한**이 있다 (예: 최대 5~10개)
- 바이브코딩으로 만들면 연결을 닫지 않는 코드가 나오기 쉽다
- 개발 중에는 사용자가 나 혼자니까 문제가 없다가, 여러 사람이 쓰기 시작하면 터진다

**올바른 연결 관리:**

```python
# ❌ 잘못된 패턴: 연결을 닫지 않음
@app.get("/api/users")
def get_users():
    db = SessionLocal()
    users = db.query(User).all()
    return users
    # db.close()가 없다! → 연결이 반환되지 않고 쌓인다

# ✅ 올바른 패턴: FastAPI의 Depends로 자동 관리
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()  # 요청이 끝나면 반드시 연결을 반환

@app.get("/api/users")
def get_users(db: Session = Depends(get_db)):
    users = db.query(User).all()
    return users
```

### 커넥션 풀 설정 확인

```python
from sqlalchemy import create_engine

# 커넥션 풀 설정 예시
engine = create_engine(
    DATABASE_URL,
    pool_size=5,        # 기본 연결 수
    max_overflow=10,    # 추가로 만들 수 있는 연결 수
    pool_timeout=30,    # 연결을 기다리는 최대 시간 (초)
    pool_recycle=1800,  # 30분마다 연결을 새로 만듦 (오래된 연결 방지)
)
```

## 4.4 마이그레이션 관련 에러

### 마이그레이션이란?

DB의 테이블 구조(스키마)를 변경하는 것을 **마이그레이션**이라고 한다. 코드에서 모델을 바꿨다고 DB가 자동으로 바뀌지 않는다.

```
코드 변경:                    DB 상태:
class User:                   users 테이블
    name = Column(String)     → name (String)  ✅ 존재
    email = Column(String)    → email?          ❌ 아직 없음!
    ↑ 새로 추가함
```

**마이그레이션을 안 하면:**
```
sqlalchemy.exc.OperationalError:
    (psycopg2.errors.UndefinedColumn) column "email" does not exist
```

### 자주 하는 실수

| 상황 | 문제 | 해결 |
|------|------|------|
| 코드에서 새 필드 추가 | DB에 컬럼이 없음 | 마이그레이션 생성 후 실행 |
| 로컬에서 마이그레이션 실행 | 배포 서버 DB에는 미반영 | 배포 후 마이그레이션도 실행 |
| 필드명 변경 | 기존 데이터 손실 가능 | 신중하게 마이그레이션 작성 |

### Alembic 기본 명령어 (SQLAlchemy 마이그레이션 도구)

```bash
# 마이그레이션 파일 생성 (코드 변경사항을 감지)
alembic revision --autogenerate -m "add email column"

# 마이그레이션 실행 (DB에 변경사항 적용)
alembic upgrade head

# 현재 DB 상태 확인
alembic current

# 마이그레이션 취소 (하나 되돌리기)
alembic downgrade -1
```

## 4.5 배포 환경에서의 DB 에러

### 확인하는 곳: Render.com Logs

배포 환경의 DB 에러도 **서버 로그에서 확인**한다. Render 대시보드 → 서비스 → Logs.

### 자주 발생하는 문제들

#### DATABASE_URL 미설정

```
# Render 환경변수에 DATABASE_URL을 안 넣으면
KeyError: 'DATABASE_URL'
# 또는
sqlalchemy.exc.ArgumentError: Could not parse rfc1738 URL from string 'None'
```

**해결:** Render 대시보드 → Environment → DATABASE_URL 환경변수 추가

#### 로컬 SQLite → 운영 PostgreSQL 전환

```python
# 로컬 개발 시 (SQLite)
DATABASE_URL = "sqlite:///./test.db"

# 배포 시 (PostgreSQL) - 문법이 다르다!
DATABASE_URL = "postgresql://user:pass@host:5432/dbname"
```

**SQLite와 PostgreSQL의 차이로 인한 에러:**

| 항목 | SQLite | PostgreSQL |
|------|--------|-----------|
| 타입 검사 | 느슨함 (문자열을 숫자에 넣어도 됨) | 엄격함 (타입이 안 맞으면 에러) |
| 대소문자 | `LIKE`가 대소문자 구분 안 함 | `LIKE`가 대소문자 구분함 |
| Boolean | 0/1로 저장 | true/false로 저장 |
| 동시 접속 | 쓰기 시 잠금 (1명만) | 여러 명 동시 가능 |

> **자주 겪는 시나리오:** 로컬(SQLite)에서는 잘 되는데 배포(PostgreSQL)하면 타입 에러가 나는 경우. "로컬에선 되는데 왜?"의 대표적 원인.

#### DB 무료 플랜 제한

Render.com 등 클라우드 DB 무료 플랜에는 제한이 있다:

| 제한 항목 | 영향 |
|----------|------|
| 최대 연결 수 (5~20개) | 커넥션 풀 부족 에러 |
| 저장 용량 (256MB~1GB) | 데이터 저장 실패 |
| 자동 만료 (90일 등) | 어느 날 갑자기 DB가 사라짐 |
| 지역 제한 | 서버와 DB가 멀면 느려짐 |

> **중요:** Render.com의 무료 PostgreSQL DB는 **90일 후 자동 삭제**된다. 갑자기 DB 연결 에러가 나면 만료 여부를 먼저 확인하자.

## 4.6 DB 에러 디버깅 체크리스트

에러가 났을 때 이 순서대로 확인한다:

```
□ 1. 서버 터미널(또는 Render Logs)에서 에러 메시지 확인
□ 2. 에러가 연결 문제인지, 쿼리 문제인지 구분
     - "connection" 관련 → 연결 문제 (4.1)
     - 테이블/컬럼 관련 → 쿼리/마이그레이션 문제 (4.2, 4.3)
□ 3. DATABASE_URL 환경변수가 올바른지 확인
□ 4. DB 서버가 살아있는지 확인 (Render 대시보드에서 DB 상태)
□ 5. 마이그레이션이 최신 상태인지 확인
□ 6. 커넥션 풀 에러라면 연결 관리 코드 점검
```

---

## 핵심 정리

1. DB 에러는 **프론트에서는 500으로만 보인다** → 반드시 서버 로그를 확인해야 진짜 원인을 알 수 있다
2. 연결 에러는 **DATABASE_URL**을 가장 먼저 의심한다
3. 커넥션 풀은 **한정된 자원**이다 — 연결을 사용 후 반드시 반환해야 한다
4. 코드에서 모델을 바꾸면 **마이그레이션**을 실행해야 DB에 반영된다
5. 로컬(SQLite)과 운영(PostgreSQL)은 **동작이 다르다** — 배포 후 반드시 테스트
6. 클라우드 무료 DB는 **연결 수, 용량, 만료 기한** 제한이 있다
