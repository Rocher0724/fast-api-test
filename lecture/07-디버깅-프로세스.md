# 7장. 에러를 체계적으로 해결하는 5단계 프로세스

## 학습 목표

에러를 만났을 때 당황하지 않고 따라갈 수 있는 **반복 가능한 루틴**을 만든다.

---

## 디버깅 5단계

```
┌─────────────────────────────────────────────┐
│  Step 1: 재현      "정확히 뭘 하면 에러가 나지?"  │
│       ↓                                      │
│  Step 2: 위치 파악  "어디에서 난 에러지?"         │
│       ↓                                      │
│  Step 3: 에러 읽기  "에러가 뭐라고 하지?"         │
│       ↓                                      │
│  Step 4: 원인 탐색  "왜 이런 에러가 나지?"        │
│       ↓                                      │
│  Step 5: 수정+확인  "고쳤고, 진짜 해결됐나?"      │
│       ↓                                      │
│  (해결 안 됐으면 Step 1로 돌아간다)              │
└─────────────────────────────────────────────┘
```

---

## Step 1: 재현

> "정확히 어떤 동작을 하면 에러가 나는가?"

에러를 **내가 원할 때 다시 발생**시킬 수 있어야 한다.

**하지 말아야 할 것:**
- "아까 에러가 났었는데..." → 정확히 어떤 동작인지 모르면 고칠 수 없다
- "가끔 에러가 나요" → 언제 나고 언제 안 나는지 조건을 찾아야 한다

**해야 할 것:**
```
✅ 구체적으로 기록
"로그인 페이지에서 이메일 없이 비밀번호만 입력하고 로그인 버튼을 누르면 에러가 난다"
"사용자 목록 페이지에 처음 접속할 때 데이터가 안 나온다"
```

---

## Step 2: 위치 파악

> "이 에러는 어디에서 발생하는가?"

에러가 발생하는 **지점**에 따라 확인해야 할 곳이 다르다.

```
증상을 보고 판단:

화면이 안 보이거나 깨짐?     → 프론트엔드       → 2장 참고
버튼 눌러도 반응 없음?       → 프론트엔드 (JS)  → 2장 참고
데이터가 안 나옴?            → 통신 or 백엔드   → Network 탭부터 확인
서버 시작이 안 됨?           → 백엔드           → 3장 참고
로컬에선 되고 배포하면 안 됨?  → 배포 환경       → 6장 참고
```

### 확인 장소 빠른 참조

| 확인할 곳 | 여는 방법 | 무엇을 보는가 |
|----------|----------|-------------|
| 브라우저 Console | F12 → Console 탭 | JS 에러 (빨간 글씨) |
| 브라우저 Network | F12 → Network 탭 | API 요청/응답, 상태 코드 |
| 서버 터미널 | uvicorn 실행 중인 터미널 | Traceback, 에러 로그 |
| FastAPI /docs | localhost:8000/docs | API를 프론트 없이 직접 테스트 |
| Render Logs | Render 대시보드 → Logs | 배포 서버의 로그 |

### 프론트 vs 백 에러 분리법

```
1. 브라우저 Network 탭을 열고 에러가 나는 동작을 수행

2. Network에 요청이 있는가?
   ├── 요청이 없다 → 프론트엔드 문제 (JS가 요청을 안 보냄)
   └── 요청이 있다 → 상태 코드 확인
       ├── 4xx → 프론트가 잘못 보냄 (URL, Method, Body 확인)
       ├── 5xx → 서버 코드 문제 (서버 터미널 Traceback 확인)
       └── 200인데 데이터가 이상 → 서버가 잘못된 데이터를 보내거나
                                   프론트가 잘못 읽고 있음

3. 확실히 모르겠으면 → /docs에서 API 직접 테스트
   ├── /docs에서도 에러 → 백엔드 문제 확실
   └── /docs에서는 정상 → 프론트엔드 문제 확실
```

## Step 3: 에러 메시지 읽기

> "에러 메시지가 정확히 뭐라고 하는가?"

**에러 메시지를 끝까지 읽는다.** 이것이 초보와 중급의 가장 큰 차이다.

### 에러 메시지에서 찾아야 할 3가지

```
1. 에러 종류 (Type)
   → TypeError, ConnectionError, IntegrityError ...
   → "어떤 종류의 문제인가"를 알려준다

2. 에러 설명 (Message)
   → "Cannot read properties of undefined"
   → "field required"
   → "connection refused"
   → 구체적으로 뭐가 잘못됐는지 알려준다

3. 위치 (Location)
   → 파일명:줄번호 (main.py:25)
   → 문제가 된 필드 (loc: ["body", "username"])
   → 정확히 어디를 고쳐야 하는지 알려준다
```

### 프론트엔드 에러 메시지 예시

```
Uncaught TypeError: Cannot read properties of undefined (reading 'name')
    at script.js:15:3

→ 에러 종류: TypeError
→ 에러 설명: undefined의 'name' 속성을 읽을 수 없다
→ 위치: script.js 파일 15번째 줄
→ 해석: 15번 줄에서 사용하는 변수가 undefined인데 .name을 읽으려 했다
```

### 백엔드 에러 메시지 예시

```
sqlalchemy.exc.IntegrityError: (psycopg2.errors.UniqueViolation)
    duplicate key value violates unique constraint "users_email_key"
    DETAIL: Key (email)=(test@test.com) already exists.

→ 에러 종류: IntegrityError (데이터 무결성 위반)
→ 에러 설명: unique 제약조건 위반 - 이메일이 중복됨
→ 위치: email 필드, 값은 test@test.com
→ 해석: 이미 같은 이메일로 가입한 사용자가 있다
```

## Step 4: 원인 탐색

> "왜 이런 에러가 나는가?"

에러 메시지를 읽어도 원인을 모르겠을 때, 다음 방법을 사용한다.

### 방법 1: 에러 메시지 검색

에러 메시지의 **핵심 부분**을 복사해서 검색한다.

```
❌ 나쁜 검색: 에러 메시지 전체를 복붙 (너무 구체적이라 결과가 없음)
❌ 나쁜 검색: "에러가 나요" (너무 추상적)

✅ 좋은 검색: 에러 종류 + 핵심 메시지
   "sqlalchemy IntegrityError UniqueViolation"
   "fastapi 422 field required"
   "TypeError Cannot read properties of undefined"
```

### 방법 2: AI에게 질문하기

AI에게 효과적으로 질문하는 공식:

```
[에러 상황] + [에러 메시지] + [관련 코드]

예시:
"FastAPI 서버에서 사용자 생성 API를 호출하면 422 에러가 납니다.

에러 메시지:
{
    "detail": [{"loc": ["body", "username"], "msg": "field required"}]
}

서버 코드:
@app.post("/api/users")
def create_user(user: UserCreate): ...

프론트 코드:
fetch("/api/users", {
    method: "POST",
    body: JSON.stringify({ user_name: "홍길동" })
})

뭐가 잘못된 건가요?"
```

**이렇게 질문하면 AI가 바로 "user_name이 아니라 username으로 보내야 합니다"라고 답해준다.**

### 방법 3: 이분법 디버깅

문제 범위를 **절반씩 줄여나가는** 방법.

```
데이터가 안 나온다
├── API 호출이 나가는가? (Network 탭)
│   ├── 안 나감 → JS 코드 문제 (fetch 자체가 실행 안 됨)
│   └── 나감 → 응답이 어떤가?
│       ├── 에러 응답 → 서버 문제 (서버 로그 확인)
│       └── 정상 응답 → 데이터를 화면에 그리는 코드 문제
│           ├── console.log(data)로 데이터 확인
│           └── DOM 조작 코드 확인
```

## Step 5: 수정 후 확인

> "고친 뒤, 처음과 같은 동작으로 다시 확인한다."

**하지 말아야 할 것:**
- "수정했으니 됐겠지" → 반드시 확인해야 한다
- 다른 동작으로 확인 → Step 1에서 기록한 **같은 동작**으로 확인해야 한다

**해야 할 것:**
```
1. 코드 수정
2. 저장
3. 서버 재시작 (필요한 경우)
4. Step 1에서 기록한 정확히 같은 동작 수행
5. 에러가 안 나면 → 해결!
6. 다른 에러가 나면 → Step 1부터 다시 (이건 새로운 에러)
7. 같은 에러가 나면 → 수정이 안 먹힌 것 (캐시? 서버 재시작 안 함? 다른 파일을 고침?)
```

### 수정이 안 먹힐 때 체크

```
□ 파일을 저장했는가?
□ 올바른 파일을 수정했는가? (같은 이름의 다른 파일 아닌지)
□ 서버를 재시작했는가? (--reload 옵션 없으면 수동 재시작 필요)
□ 브라우저 캐시가 남아있지 않은가? (Cmd+Shift+R로 강력 새로고침)
□ 배포 환경이면 재배포했는가?
```

## 실전 연습: 이 5단계를 적용해보자

### 시나리오

> "회원가입 버튼을 누르면 화면에 '에러가 발생했습니다'만 나오고 가입이 안 된다"

### 5단계 적용

```
Step 1: 재현
→ 회원가입 페이지에서 이름, 이메일, 비밀번호를 입력하고 가입 버튼을 누른다

Step 2: 위치 파악
→ F12 → Network 탭 확인
→ POST /api/signup 요청이 보이고, 상태 코드가 422
→ 서버에 요청은 갔지만, 서버가 422로 거부한 것 → 프론트가 보낸 데이터 문제

Step 3: 에러 읽기
→ Network 탭 → Response 확인
→ {"detail": [{"loc": ["body", "password"], "msg": "ensure this value has at least 8 characters"}]}
→ 해석: password가 최소 8자 이상이어야 하는데 짧게 보냈다

Step 4: 원인 탐색
→ 프론트에서 비밀번호 길이 체크를 안 하고 보냈거나
→ 서버의 최소 길이 검증이 있다는 걸 몰랐거나

Step 5: 수정+확인
→ 프론트에 비밀번호 길이 안내 추가 + 8자 이상으로 테스트
→ 같은 동작으로 회원가입 → 성공!
```

## 핵심 정리

1. **재현** → 에러를 다시 발생시킬 수 있어야 고칠 수 있다
2. **위치 파악** → Console, Network, 서버 터미널, /docs를 활용해서 어디 문제인지 분리
3. **에러 읽기** → 에러 메시지의 종류, 설명, 위치 3가지를 찾는다
4. **원인 탐색** → 에러 메시지 검색, AI 질문(상황+에러+코드), 이분법 디버깅
5. **수정+확인** → 고친 뒤 반드시 같은 동작으로 재확인. 캐시와 재시작 주의
