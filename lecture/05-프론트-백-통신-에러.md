## 학습 목표

프론트엔드(브라우저)와 백엔드(FastAPI 서버) 사이의 통신에서 발생하는 에러를 **어디서 확인하고** 해결하는 방법을 배운다.

## 5.1 CORS 에러

### 에러는 어디서 확인하는가?

**브라우저 Console 탭**에서 확인한다. 아래처럼 빨간 글씨가 나온다:

```
❌ Access to fetch at 'https://myapp.onrender.com/api/users'
   from origin 'https://mysite.com' has been blocked by CORS policy:
   No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

### CORS가 뭔가?

브라우저에는 **보안 규칙**이 있다: "다른 주소의 서버에 요청을 보내면 차단한다."

```
프론트엔드 주소:  https://mysite.com
서버 주소:       https://myapp.onrender.com

→ 주소가 다르다 → 브라우저가 차단! → CORS 에러
```

**왜 이런 규칙이 있는가?** 악성 사이트가 사용자 모르게 다른 서버에 요청을 보내는 것을 막기 위해서다. 하지만 우리의 프론트엔드와 서버는 같은 서비스이므로, 서버에서 "이 주소에서 오는 요청은 허용해"라고 설정해줘야 한다.

### FastAPI에서 CORS 설정하는 법

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

# CORS 허용 설정
app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5500",          # 로컬 개발용 (Live Server)
        "http://127.0.0.1:5500",          # 로컬 개발용
        "https://mysite.com",             # 배포된 프론트엔드 주소
    ],
    allow_credentials=True,
    allow_methods=["*"],        # 모든 HTTP 메서드 허용 (GET, POST, PUT, DELETE)
    allow_headers=["*"],        # 모든 헤더 허용
)
```

### 왜 "로컬에서는 되고 배포하면 안 되는가?"

```python
# 로컬 개발할 때
allow_origins=["http://localhost:5500"]    # ← 로컬 주소만 허용해놓음

# 배포하면
# 프론트가 https://mysite.com에서 실행됨
# → allow_origins에 없음 → CORS 에러!

# 해결: 배포 주소도 추가
allow_origins=[
    "http://localhost:5500",
    "https://mysite.com",        # ← 이걸 추가해야 함
]
```

> **주의:** `allow_origins=["*"]`은 모든 주소를 허용하는 설정이다. 개발 중에는 편하지만, 실제 운영에서는 보안상 **특정 도메인만 허용**하는 것이 좋다.

### CORS 에러 디버깅 순서

```
1. 브라우저 Console에서 CORS 에러 메시지 확인
2. 에러 메시지에서 "from origin 'xxx'" 부분 확인 → 이게 프론트엔드 주소
3. 서버 코드에서 allow_origins에 그 주소가 포함되어 있는지 확인
4. 없으면 추가 → 서버 재시작 (또는 재배포)
```

## 5.2 fetch() / API 호출 실수

### 에러는 어디서 확인하는가?

**브라우저 Network 탭** + **Console 탭** 양쪽을 확인한다.

### URL 경로 오타

```javascript
// 서버에 등록된 경로
// @app.get("/api/users")

// ❌ 오타들
fetch("/api/user")           // 's' 빠짐 → 404
fetch("/api/Users")          // 대문자 → 404 (URL은 대소문자 구분)
fetch("api/users")           // 앞에 '/' 빠짐 → 잘못된 경로
fetch("/api/users/")         // 뒤에 '/' 추가 → 경우에 따라 404

// ✅ 정확한 경로
fetch("/api/users")
```

**확인 방법:** Network 탭에서 **Request URL**을 확인하고, 서버 코드의 라우트와 정확히 비교한다.

### HTTP Method 불일치

```python
# 서버: POST만 받는다
@app.post("/api/users")
def create_user(user: UserCreate):
    ...
```

```javascript
// ❌ GET으로 보내면 → 405 Method Not Allowed
fetch("/api/users")

// ✅ POST로 보내야 함
fetch("/api/users", {
    method: "POST",
    ...
})
```

**확인 방법:** Network 탭에서 **Request Method**를 확인한다.

### Request Body 형식 오류

```javascript
// ❌ Content-Type 헤더 누락 → 서버가 JSON으로 인식 못 함 → 422
fetch("/api/users", {
    method: "POST",
    body: JSON.stringify({ username: "홍길동", age: 25 })
});

// ✅ Content-Type 헤더를 반드시 포함
fetch("/api/users", {
    method: "POST",
    headers: {
        "Content-Type": "application/json"    // ← 이게 있어야 서버가 JSON으로 읽음
    },
    body: JSON.stringify({ username: "홍길동", age: 25 })
});
```

### async/await 빠뜨리기

```javascript
// ❌ await를 안 쓰면 → Promise 객체가 출력됨
function loadUsers() {
    const response = fetch("/api/users");
    console.log(response);   // Promise { <pending> } ← 데이터가 아님!
}

// ✅ async/await를 사용
async function loadUsers() {
    const response = await fetch("/api/users");
    const data = await response.json();    // .json()에도 await 필요!
    console.log(data);       // 실제 데이터 출력
}
```

> **알아두면 좋은 것:** `fetch()`는 **비동기 함수**다. 서버에 요청을 보내고 응답을 기다리는 동안 다른 코드가 먼저 실행된다. `await`는 "응답이 올 때까지 여기서 기다려"라는 의미다. 이걸 빠뜨리면 응답이 오기 전에 데이터를 사용하려 해서 에러가 난다.

## 5.3 JSON 데이터 주고받기 실수

### 에러는 어디서 확인하는가?

**브라우저 Network 탭**의 **Payload(보낸 것)**와 **Response(받은 것)**를 비교한다.

### 보낼 때 실수

```javascript
// ❌ JSON.stringify() 안 하고 보내기
fetch("/api/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: { username: "홍길동" }     // ← 객체를 그대로 보내면 "[object Object]"가 전송됨
});

// ✅ 반드시 JSON.stringify()로 변환
fetch("/api/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ username: "홍길동" })
});
```

### 받을 때 실수

```javascript
// ❌ .json()을 안 하고 사용
const response = await fetch("/api/users");
console.log(response);         // Response 객체가 나옴 (데이터가 아님!)
console.log(response.name);    // undefined

// ✅ .json()으로 변환해야 실제 데이터를 얻는다
const response = await fetch("/api/users");
const data = await response.json();
console.log(data);             // 실제 데이터
console.log(data.name);       // "홍길동"
```

### 필드명 불일치

이건 **가장 찾기 어려운 에러** 중 하나다. 에러 없이 `undefined`가 나오기 때문이다.

```python
# 서버가 보내는 데이터
@app.get("/api/users/1")
def get_user():
    return {"user_name": "홍길동", "user_age": 25}
#           ─────────             ────────
#           snake_case (파이썬 스타일)
```

```javascript
// 프론트에서 받는 데이터
const data = await response.json();
console.log(data.userName);    // undefined ← camelCase로 접근했기 때문
console.log(data.user_name);   // "홍길동" ← 서버가 보낸 필드명과 정확히 일치해야 함
```

**확인 방법:**
1. Network 탭 → Response에서 **서버가 보낸 실제 데이터**를 확인
2. 내 JS 코드에서 **접근하는 필드명**과 비교
3. `console.log(data)`로 **전체 데이터 구조**를 먼저 확인하는 습관

## 5.4 에러 응답 처리를 안 하면 생기는 문제

```javascript
// ❌ 에러 처리를 안 하면: 서버가 에러를 보내도 모른다
async function loadUsers() {
    const response = await fetch("/api/users");
    const data = await response.json();    // 에러 응답도 JSON으로 파싱 시도
    renderUsers(data);                      // 에러 데이터로 렌더링 → 화면이 깨짐
}

// ✅ 에러 처리를 하면: 뭐가 잘못됐는지 알 수 있다
async function loadUsers() {
    const response = await fetch("/api/users");

    if (!response.ok) {
        // 에러 응답이면 여기서 처리
        const errorData = await response.json();
        console.error("서버 에러:", response.status, errorData);
        alert("데이터를 불러오는데 실패했습니다");
        return;
    }

    const data = await response.json();
    renderUsers(data);
}
```

> **알아두면 좋은 것:** `fetch()`는 네트워크 에러(서버가 아예 응답 안 함)일 때만 에러를 throw한다. 서버가 404나 500을 보내도 `fetch()`는 성공으로 취급한다. 그래서 반드시 `response.ok`로 확인해야 한다.

---

## 핵심 정리

1. **CORS 에러** → Console 탭에서 확인 → 서버의 `allow_origins`에 프론트 주소 추가
2. **404/405 에러** → Network 탭에서 URL과 Method 확인 → 서버 라우트와 비교
3. **422 에러** → Network 탭 Response의 `detail`에서 어떤 필드가 잘못됐는지 확인
4. `fetch()` 사용 시 3가지 필수: `Content-Type` 헤더, `JSON.stringify()`, `async/await`
5. 데이터가 `undefined`이면 **필드명 불일치**를 의심 → `console.log(data)`로 실제 구조 확인
6. `response.ok`로 에러 응답을 반드시 체크해야 한다
