# 10장. 라이브 디버깅 데모 시나리오

## 사용법

강의 중 실제로 에러를 만들고 고치는 과정을 시연한다.
각 시나리오는 **일부러 에러를 발생시키고 → 찾고 → 고치는** 흐름으로 진행한다.

---

## 데모 1: 프론트엔드 에러 (3분)

### 설정: 일부러 에러를 만든다

```javascript
// static/script.js
async function loadUsers() {
    const response = await fetch("/api/users");
    const data = await response.json();

    // 일부러 잘못된 필드명 사용
    document.getElementById("user-list").innerHTML = data.userName;  // ← 에러!
}
loadUsers();
```

### 시연 순서

```
1. 브라우저에서 페이지를 연다 → 데이터가 안 보임
2. "자, 에러가 났습니다. 어디를 볼까요?"
3. F12 → Console 탭 열기
4. 에러 메시지 읽기:
   "Cannot read properties of undefined (reading 'innerHTML')"
   또는 데이터가 undefined
5. console.log(data) 추가하여 실제 데이터 구조 확인
6. data가 배열이므로 data[0].name으로 수정
7. 새로고침 → 해결!

포인트: Console 탭에서 에러를 확인하고, console.log()로 값을 추적하는 패턴
```

## 데모 2: 422 에러 (3분)

### 설정: 필드명을 일부러 틀리게 보낸다

```javascript
// 프론트에서 보내는 코드
fetch("/api/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
        user_name: "홍길동",     // ← 서버는 'username'을 기대
        age: "스물다섯"           // ← 서버는 숫자를 기대
    })
});
```

```python
# 서버 코드
class UserCreate(BaseModel):
    username: str     # 'user_name'이 아닌 'username'
    age: int          # 문자열이 아닌 정수

@app.post("/api/users")
def create_user(user: UserCreate):
    return {"message": f"{user.username} 생성 완료"}
```

### 시연 순서

```
1. 가입 버튼 클릭 → 실패
2. "어디를 볼까요? 일단 Network 탭"
3. Network 탭 → POST /api/users 클릭 → Status: 422
4. Response 탭 확인:
   {
     "detail": [
       {"loc": ["body", "username"], "msg": "field required"},
       {"loc": ["body", "age"], "msg": "value is not a valid integer"}
     ]
   }
5. "422의 detail을 읽으면 뭐가 잘못됐는지 다 알려줍니다"
6. 필드명 수정 (user_name → username) + 타입 수정 ("스물다섯" → 25)
7. 다시 시도 → 성공!

포인트: 422의 Response에서 detail을 읽는 습관, /docs에서도 테스트해보기
```

## 데모 4: 서버 500 에러 (3분)

### 설정: 서버 코드에 일부러 버그를 넣는다

```python
@app.get("/api/users/{user_id}")
def get_user(user_id: int):
    users = {"1": "홍길동", "2": "김철수"}
    return {"name": users[str(user_id + 100)]}   # ← KeyError 발생!
```

### 시연 순서

```
1. /api/users/1 호출 → 500 에러
2. "500이니까 서버 문제입니다. 어디를 볼까요?"
3. 서버 터미널 확인 → Traceback 출력됨
4. "Traceback은 아래에서 위로 읽습니다"
5. 맨 아래: KeyError: '101'
6. 바로 위: main.py line XX → 해당 줄 확인
7. users[str(user_id + 100)]이 잘못됨 → users[str(user_id)]로 수정
8. 서버 재시작 → 성공!

포인트: 500 에러는 서버 터미널에서 Traceback을 읽어야 한다
```

## 데모 진행 팁

### 준비물

```
□ 로컬 FastAPI 서버가 실행 가능한 환경
□ 에러 버전과 정상 버전의 코드를 미리 준비 (빠른 전환용)
□ 브라우저 DevTools 화면을 크게 보여줄 수 있는 화면 설정
□ Render 대시보드 접근 가능
```

### 진행 요령

```
1. 에러를 보여주기 전에 "어디를 봐야 할까요?" 질문
2. 수강생이 추측하게 한 다음 DevTools를 열어서 확인
3. 에러 메시지를 같이 읽으면서 해석
4. 고친 후 반드시 같은 동작으로 재확인 (Step 5 강조)
5. "이 패턴을 기억하세요"로 마무리
```

### 시간 관리

| 데모 | 시간 | 강조 포인트 |
|------|------|-----------|
| 1. 프론트엔드 에러 | 3분 | Console 탭 + console.log() |
| 2. 422 에러 | 3분 | Network 탭 + Response detail |
| 3. CORS 에러 | 3분 | Console + 서버 설정 |
| 4. 500 에러 | 3분 | 서버 터미널 + Traceback 읽기 |
| 5. 배포 에러 | 2분 | Render Logs |
| **합계** | **14분** | |

> 강의 시간이 부족하면 데모 2(422)와 데모 4(500)만 해도 핵심은 전달된다. 이 두 가지가 **가장 자주 만나는 에러**다.
