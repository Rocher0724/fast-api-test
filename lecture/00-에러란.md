# 0장. 에러란?

> 이 강의를 시작하기 전에, 우리가 앞으로 수없이 마주칠 **"에러"**가 정확히 무엇인지 짚고 넘어가자.

---

## 에러의 정의

**에러란, 문제를 수정하기 위한 안내문이다.**

어떤 것이 잘못되었는지 알리기 위한 **합의된 안내문**이라고 할 수 있다.

컴퓨터는 우리에게 화를 내는 게 아니다. 에러 메시지는 **"여기가 이상해요, 이렇게 고쳐주세요"** 라는 친절한 길 안내와 같다.

---

## 에러를 대하는 자세

많은 초보 개발자가 에러를 보면 **겁부터 먹는다.**
빨간 글씨, 영어, 긴 로그... 당연히 무섭다.

하지만 에러는 적이 아니라 **동료**다.

| 에러가 없을 때 | 에러가 있을 때 |
|---|---|
| 뭐가 잘못된 건지 모른다 | 정확히 뭐가 잘못됐는지 알려준다 |
| 어디를 고쳐야 할지 감이 없다 | 파일명, 줄 번호까지 알려준다 |
| 멀쩡해 보이는데 안 된다 | 원인을 추적할 단서가 있다 |

**진짜 무서운 건 에러가 안 나는데 안 되는 상황이다.**
에러가 나온다는 건, 적어도 빠르게 고칠 가능성이 있다는 뜻이다.

---

## 에러의 3가지 구성 요소

대부분의 에러 메시지는 다음 세 가지를 포함한다:

```
1. 어디서 (Where)  → 파일명, 줄 번호, 함수명
2. 무엇이 (What)   → 어떤 종류의 문제인지 (TypeError, 404, SyntaxError...)
3. 왜 (Why)        → 구체적인 원인 설명
```

예를 들어:

```
File "main.py", line 15, in get_user     ← 어디서
TypeError:                                ← 무엇이
  string indices must be integers         ← 왜
```

이 세 가지만 읽을 수 있으면, 에러의 절반은 이미 해결한 것이다.

---

## 에러는 언어다

개발자들 사이에서 에러 메시지는 일종의 **공용어**다.

- `404` → "못 찾겠다"
- `500` → "서버가 터졌다"
- `422` → "보낸 데이터가 이상하다"
- `CORS error` → "다른 곳에서 요청해서 막혔다"

이 "언어"를 하나씩 익히면, 에러가 나왔을 때 메시지만 보고도 어디를 열어봐야 하는지 감이 온다. 이 강의는 바로 그 감을 만들어주는 과정이다.
