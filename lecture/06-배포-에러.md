# 6장. 배포 에러 확인법: Render.com

## 학습 목표

로컬에서는 잘 되는데 배포하면 안 되는 상황에서 **어디서 에러를 확인하고**, 원인을 찾는 방법을 배운다. 배포 환경의 특수성도 이해한다.

---

## 6.1 Render 대시보드에서 로그 보기

### 어디서 확인하는가?

Render 대시보드 → 서비스 선택 → **"Logs"** 탭

이것이 배포 환경에서의 **유일한 에러 확인 수단**이다. 로컬의 터미널 창 역할을 한다.

### 두 종류의 로그

#### Deploy 로그 (배포 로그)

코드를 올렸을 때 서버를 **빌드하고 시작하는 과정**의 로그.

```
==> Cloning from https://github.com/myuser/myapp...
==> Using Python 3.11.5
==> Installing dependencies with pip...
Collecting fastapi>=0.115.0
Collecting uvicorn[standard]>=0.34.0
Collecting sqlalchemy>=2.0.0
ERROR: Could not find a version that satisfies the requirement some-package
#     ──────────────────────────────────────────────────────────────────────
#     이런 에러가 나면 배포 자체가 실패한다
```

**배포 로그에서 에러가 나면:** 서버가 아예 시작되지 않는다. 코드가 올라가지 않은 상태.

#### Runtime 로그 (실행 로그)

서버가 시작된 후 **실행 중에 발생하는** 로그. 로컬의 터미널에서 보던 것과 동일하다.

```
INFO:     Application startup complete.
INFO:     10.0.0.1 - "GET /health HTTP/1.1" 200 OK
ERROR:    Exception in ASGI application
Traceback (most recent call last):
  File "/app/main.py", line 25, in get_users
    ...
ConnectionError: Database connection failed
```

**런타임 로그에서 에러가 나면:** 서버는 돌고 있지만 특정 기능에서 에러가 발생한 것.

### 로그 구분 요약

| 상황 | 어떤 로그를 보는가 | 증상 |
|------|-------------------|------|
| 배포 후 사이트가 아예 안 열림 | Deploy 로그 | 서버 시작 실패 |
| 사이트는 열리는데 특정 기능 에러 | Runtime 로그 | 코드 실행 중 에러 |
| 배포 버튼 눌렀는데 실패 표시 | Deploy 로그 | 빌드 또는 시작 실패 |

## 6.2 배포 시 자주 발생하는 에러들

### requirements.txt에 패키지 빠뜨림

```
# Deploy 로그에서 이렇게 나온다
ERROR: No matching distribution found for some-package

# 또는 Runtime 로그에서
ModuleNotFoundError: No module named 'sqlalchemy'
```

**원인:** 로컬에서는 `pip install`로 패키지를 설치했지만, `requirements.txt`에 적지 않았다.

**확인 및 해결:**
```bash
# 로컬에서 현재 설치된 패키지 목록 확인
pip freeze

# requirements.txt에 빠진 패키지가 없는지 비교
# 빠진 게 있으면 추가한 후 다시 배포
```

> **팁:** 새 패키지를 설치할 때마다 `requirements.txt`에도 바로 추가하는 습관을 들이자.

### Start Command 설정 오류

Render에서 서버를 어떻게 시작할지 알려주는 명령어가 **Start Command**다.

```bash
# ✅ 올바른 Start Command
uvicorn app.main:app --host 0.0.0.0 --port $PORT

# ❌ 흔한 실수들
uvicorn app.main:app                           # --host와 --port가 없음
uvicorn main:app --host 0.0.0.0 --port $PORT   # 경로가 'app.main'이 아닌 'main'
uvicorn app.main:app --host 0.0.0.0 --port 8000  # $PORT 대신 8000 고정
```

**왜 `--host 0.0.0.0`이 필요한가?**
- 기본값은 `127.0.0.1`(자기 자신만 접속 가능)
- `0.0.0.0`으로 해야 **외부에서 접속 가능**

**왜 `--port $PORT`인가?**
- Render가 서버에 포트 번호를 **환경변수로 전달**한다
- 고정 포트(8000)를 쓰면 Render가 제공한 포트와 안 맞을 수 있다

### 환경변수 미설정

```python
# 코드에서 환경변수를 사용하는 경우
import os
DATABASE_URL = os.environ["DATABASE_URL"]     # 환경변수가 없으면 KeyError
SECRET_KEY = os.environ.get("SECRET_KEY")     # 없으면 None (에러는 안 나지만 문제)
```

**확인 방법:**

```
로컬: .env 파일에 값이 있다
배포: Render 대시보드 → Environment → 환경변수가 등록되어 있는지 확인
```

| 환경변수 | 로컬 (.env) | Render (Environment) |
|---------|------------|---------------------|
| DATABASE_URL | ✅ sqlite:///./test.db | ❓ 추가했는지 확인! |
| SECRET_KEY | ✅ my-secret-key | ❓ 추가했는지 확인! |

> **중요:** `.env` 파일은 `.gitignore`에 포함되어 있으므로 **GitHub에 올라가지 않는다.** 따라서 Render가 자동으로 읽을 수 없다. **반드시 Render 대시보드에서 수동 등록**해야 한다.

### Python 버전 불일치

```
# Deploy 로그에서
SyntaxError: invalid syntax
# Python 3.11 문법을 썼는데 Render가 3.9를 쓰는 경우

# 또는
ModuleNotFoundError: No module named 'tomllib'
# tomllib은 Python 3.11부터 추가됨
```

**해결:** Render 대시보드에서 Python 버전을 지정하거나, 프로젝트 루트에 `runtime.txt` 생성:
```
python-3.11.5
```

## 6.3 "로컬 vs 배포" 환경 차이 체크리스트

로컬에서 되는데 배포에서 안 되는 건 대부분 **환경 차이** 때문이다.

### 파일 경로 대소문자

```python
# ❌ Mac에서는 되지만 Linux(Render)에서는 에러
from app.Models import User     # 파일명이 models.py인데 Models로 import

# ✅ 정확한 대소문자
from app.models import User
```

**왜?** Mac의 파일 시스템(APFS)은 대소문자를 **구분하지 않지만**, Linux는 **구분한다.** Render의 서버는 Linux다.

### 환경 차이 종합 체크리스트

```
□ 환경변수가 Render에도 등록되어 있는가?
□ requirements.txt에 모든 패키지가 포함되어 있는가?
□ 파일명/import 경로의 대소문자가 정확한가?
□ Start Command가 올바른가?
□ Python 버전이 맞는가?
□ DB 연결 주소가 배포용인가? (localhost 아닌지)
□ CORS 설정에 배포 도메인이 포함되어 있는가?
□ 하드코딩된 경로(/Users/myname/...)가 없는가?
```

## 6.4 Render 무료 플랜 콜드 스타트

### 이게 뭔가?

Render 무료 플랜은 **15분간 요청이 없으면 서버를 잠재운다(Sleep).** 그리고 다음 요청이 오면 서버를 다시 깨우는데, 이 과정에서 **30초~1분** 걸린다. 이걸 **콜드 스타트(Cold Start)**라고 한다.

```
사용자 접속 → "사이트가 안 열려요!" → 30초~1분 후 → "아, 열리네요"
              ─────────────────────
              이건 에러가 아니라 콜드 스타트
```

### 에러처럼 보이는 증상

| 증상 | 실제 원인 |
|------|----------|
| 사이트 접속 시 오래 로딩됨 | 서버가 잠에서 깨어나는 중 |
| 첫 API 호출이 타임아웃 | 서버 기동 중 요청이 들어옴 |
| "Service Unavailable (503)" | 서버가 아직 준비 안 됨 |
| 두 번째 접속부터는 빠름 | 서버가 이미 깨어있음 |

### 어디서 확인하는가?

Render 대시보드 → 서비스 상태를 확인:
- **"Sleeping"** 표시 → 서버가 잠든 상태
- **"Live"** 표시 → 서버가 동작 중

### 대응 방법

```
1. 이건 에러가 아니라 무료 플랜의 특성임을 이해한다
2. 사용자에게 "첫 접속 시 30초 정도 걸릴 수 있다"고 안내한다
3. 해결하고 싶으면:
   - 유료 플랜으로 업그레이드 (서버가 안 잠듦)
   - 외부 서비스로 주기적 ping (예: UptimeRobot으로 14분마다 /health 호출)
   - 하지만 Render가 이를 감지하면 제한할 수 있으니 주의
```

> **핵심:** "아침에 처음 접속하면 느리고, 이후엔 빠르다"면 콜드 스타트다. 코드를 고칠 필요 없다.

## 6.5 환경변수 관리: .env와 python-dotenv

### 환경변수란?

코드에 직접 쓰면 안 되는 값(비밀번호, API 키, DB 주소 등)을 **코드 바깥에서 주입**하는 방법이다.

```python
# ❌ 코드에 비밀번호를 직접 쓰면
DATABASE_URL = "postgresql://user:mypassword123@server:5432/db"  # GitHub에 올라감!

# ✅ 환경변수로 가져오면
import os
DATABASE_URL = os.environ.get("DATABASE_URL")  # 코드에 비밀번호가 없음
```

### 로컬에서 환경변수 관리: .env + python-dotenv

로컬 개발 시에는 `.env` 파일에 환경변수를 저장하고, `python-dotenv`로 읽는다.

**1단계: python-dotenv 설치**
```bash
pip install python-dotenv
```
`requirements.txt`에도 추가:
```
python-dotenv>=1.0.0
```

**2단계: .env 파일 생성** (프로젝트 루트)
```
DATABASE_URL=sqlite:///./test.db
SECRET_KEY=my-local-secret-key
DEBUG=true
```

**3단계: 코드에서 불러오기**
```python
from dotenv import load_dotenv
import os

load_dotenv()  # .env 파일의 값을 환경변수로 로드

DATABASE_URL = os.environ.get("DATABASE_URL")
SECRET_KEY = os.environ.get("SECRET_KEY")
```

**4단계: .gitignore에 .env 추가** (GitHub에 올라가면 안 됨!)
```
.env
```

### 로컬과 배포의 환경변수 흐름

```
로컬 개발:  .env 파일  →  python-dotenv가 읽음  →  os.environ으로 접근
배포:       Render 대시보드 Environment  →  자동으로 os.environ에 설정됨

코드는 동일: os.environ.get("DATABASE_URL")
```

> **핵심:** 코드는 항상 `os.environ`에서 읽는다. 로컬에서는 `.env`에서, 배포에서는 Render 설정에서 값이 공급될 뿐이다.

## 6.6 알아두면 좋은 것: 실제 서버 운영

### Render.com은 편의 도구다

Render는 "서버 관리를 대신 해주는 서비스"다. 실제로는 이런 일이 뒤에서 일어난다:

```
Render가 해주는 것:
1. Linux 서버(컨테이너) 생성
2. GitHub에서 코드 다운로드
3. 패키지 설치 (pip install -r requirements.txt)
4. Start Command 실행
5. HTTPS 인증서 설정
6. 로그 수집 및 표시
```

### 나중에 직접 서버를 운영하면?

AWS, GCP, 또는 직접 서버를 운영하면 위 과정을 **직접** 해야 한다:

```bash
# 1. 서버에 SSH 접속
ssh user@12.34.56.78

# 2. 코드 다운로드
git pull origin main

# 3. 패키지 설치
pip install -r requirements.txt

# 4. 서버 실행 (백그라운드)
nohup uvicorn app.main:app --host 0.0.0.0 --port 8000 &

# 5. 로그 확인
tail -f nohup.out                       # 실시간 로그
cat nohup.out | grep "ERROR"            # 에러만 필터
journalctl -u myapp -f                  # systemd 사용 시

# 6. 서버 상태 확인
ps aux | grep uvicorn                   # 프로세스가 살아있는지
curl http://localhost:8000/health       # 서버가 응답하는지
```

**지금은 Render 대시보드의 Logs만 볼 줄 알면 된다.** 하지만 이 로그가 결국 서버 내부의 텍스트 기록이라는 점을 이해하면, 나중에 어떤 환경에서든 디버깅할 수 있다.

### 배포 에러 디버깅의 핵심 원칙

> **"로컬과 배포의 차이점을 찾아라."**
>
> 코드는 같은데 환경이 다르다. 에러가 나면 "로컬과 배포에서 **뭐가 다른지**"를 체크리스트로 점검하면 대부분 해결된다.

---

## 핵심 정리

1. 배포 에러는 **Render 대시보드 → Logs 탭**에서 확인한다
2. **Deploy 로그** = 빌드/시작 실패, **Runtime 로그** = 실행 중 에러
3. 가장 흔한 원인: requirements.txt 누락, 환경변수 미설정, Start Command 오류
4. Mac과 Linux의 **대소문자 구분** 차이를 항상 주의
5. `.env` 파일은 Git에 안 올라가므로 **Render에 수동 등록** 필요
6. Render가 해주는 일의 본질을 이해하면 어떤 배포 환경에서도 디버깅 가능
